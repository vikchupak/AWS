# IAM DB Authentication

AWS IAM Database Authentication is a feature for **Amazon RDS** and **Aurora** that allows you to authenticate to your database using IAM users or roles instead of a traditional static password.

Instead of a long-lived secret, you use a short-lived **authentication token** generated by AWS Signature Version 4.

---

### How It Works

The authentication process moves the "source of truth" for identity from the database engine itself to AWS IAM.

1. **Request Token:** Your application (running on EC2, Lambda, or a local machine) uses the AWS SDK or CLI to call the `generate-db-auth-token` API.
2. **Generate Token:** AWS returns a signed, unique string of characters. This token is **valid for 15 minutes**.
3. **Connect:** Your application connects to the RDS instance using a standard database driver (like `pg` for Postgres or `mysql2` for Node.js).
4. **Token as Password:** You pass the generated token into the **password field** of your connection string.
5. **Validation:** The RDS instance recognizes the request is an IAM request, decodes the token, and verifies with IAM that the requester has the `rds-db:connect` permission.

---

### Key Benefits

* **No Password Management:** You don't have to rotate database passwords or store them in configuration files or environment variables.
* **IAM Integration:** You can use **IAM Roles** (e.g., an EC2 Instance Profile or Lambda Execution Role), which eliminates the need for hardcoded AWS credentials.
* **Security:** All connections are required to use **SSL/TLS** encryption.
* **Centralized Access:** Permissions are managed via IAM policies, making it easier to revoke access instantly without modifying the database.

---

### Implementation Essentials

To set this up, you need to coordinate three different areas:

| Component | Action Required |
| --- | --- |
| **RDS Instance** | Enable "IAM DB Authentication" in the database settings. |
| **Database Engine** | Create a user account mapped to IAM (e.g., `CREATE USER 'iam_user' IDENTIFIED WITH AWSAuthenticationPlugin AS 'RDS';`). |
| **IAM Policy** | Grant the `rds-db:connect` action on the specific database resource ARN. |

---

### Constraints & Limitations

While powerful, IAM authentication isn't a "silver bullet" for every workload:

* **Connection Limits:** It is designed for approximately **200 new connections per second**. If your application frequently opens/closes connections (high churn), you may hit throttling limits.
* **RDS Proxy:** If you use **RDS Proxy**, it can handle the IAM authentication on your behalf, which is highly recommended for serverless applications (like Lambda) to avoid connection overhead.
* **Token Expiry:** Since tokens last only 15 minutes, long-lived connections are fine, but if the connection drops after the token expires, you must generate a new one to reconnect.

### A Quick Example (Node.js)

```javascript
import { Signer } from "@aws-sdk/rds-signer";
import { Client } from "pg";

const signer = new Signer({
  region: "us-east-1",
  hostname: "your-db-instance.xyz.us-east-1.rds.amazonaws.com",
  port: 5432,
  username: "iam_db_user"
});

// Generate the token
const token = await signer.getAuthToken();

// Use the token as the password
const client = new Client({
  host: "your-db-instance.xyz.us-east-1.rds.amazonaws.com",
  user: "iam_db_user",
  password: token,
  port: 5432,
  ssl: { rejectUnauthorized: false } // SSL is mandatory
});

await client.connect();
```
